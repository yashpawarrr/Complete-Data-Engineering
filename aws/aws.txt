In Amazon Web Services (AWS), an ACL (Access Control List) is a set of rules that controls who can access resources and what actions they can perform.

<<<< ACL >>>>
There are two main types of ACLs in AWS, but most commonly people mean Network ACL (NACL) when talking about EC2 networking.

🔐 1. Network ACL (NACL)

A Network ACL works at the subnet level inside a VPC.

🧠 Key features:
Controls inbound and outbound traffic
Works as a firewall for subnets
Stateless → every request and response must be allowed explicitly
Uses rules with numbers (priority order)

⚠️ Important:
You must allow both inbound and outbound traffic
If inbound allows traffic but outbound blocks it → connection fails

🧱 2. Security Group vs NACL (Very important)

Students often confuse these:

Feature	Security Group	NACL
Level	Instance level	Subnet level
State	Stateful	Stateless
Rules	Only allow	Allow + Deny
Evaluation	All rules	Rule order (numbered)


📦 3. S3 ACL (Object-level permissions)

ACLs are also used in Amazon S3 (Simple Storage Service).

Used for:
Controlling access to buckets and objects
Granting permissions to:
AWS accounts
Public users (not recommended)
Example:
Make a file public (read-only)
Give another AWS account access


🚀 When to use NACL?
Add an extra layer of security
Block specific IP ranges
Enforce strict subnet-level control

❄️ AWS Snowball

AWS Snowball is a data transfer service used to move large amounts of data (TBs–PBs) into or out of AWS using a physical device.

🧠 Why it’s used:
Internet transfer is too slow or unreliable
You have huge data (like backups, logs, media files)


🔗 How Snowball and S3 work together
📦 Flow:
You request a Snowball device from AWS
AWS ships the device to you
You copy your data into the Snowball device
Ship it back to AWS
AWS uploads that data into an S3 bucket

👉 So:

Snowball = transport device
S3 bucket = final storage location
